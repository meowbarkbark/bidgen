## 1. 목적

6조의 BidGen 프로젝트가 해결하려는 문제, 현재 구현 범위, 입력·판단·출력 구조, 진행 상황, 미해결 문제와 다음 계획을 발표와 A4 PDF 출력에 모두 적합한 단일 HTML로 정리한다.

## 2. 핵심 소개 문구

제목은 `BidGen — 원가검증에서 공고문 초안까지`로 한다.

기존 원가계산서와 입찰공고문을 복사·수정하는 과정에서 발생하는 금액, 수식, 부가세, 날짜와 법령 누락을 줄이는 서비스로 소개한다. 원가계산서와 기준자료를 업로드하면 계산 구조와 적용 근거를 자동 검증하고, 검증된 데이터를 향후 공고문 초안 생성까지 연결하는 것이 전체 제품 비전임을 설명한다.

현재 구현 범위는 AI를 사용하지 않는 원가계산서 규칙 기반 검증과 근거 중심 결과 확인이다. 공고문 생성, 관련 법령 수집·해석은 구현하지 않은 후속 단계임을 분명히 표시한다. 필요하면 향후 AI API를 모호한 항목명·조건 설명과 법령 요약에 보조적으로 사용할 수 있지만, 숫자와 오류 여부의 최종 판정은 재현 가능한 규칙 엔진이 담당한다.

## 4. 사실관계

### 현재 입력

- 필수: 발주 유형, 원가계산서 Excel
- 기준자료: 사용자가 별도로 첨부한 외부 제비율표 Excel, 외부 노임단가표 Excel
- 선택: 표준품셈·지침 PDF

표준품셈 PDF는 현재 파일 메타데이터만 보관하고 실제 판단에는 적용하지 않는다고 표시한다.

원가계산서 통합문서 안에 제비율표·노임단가표 시트가 포함되어 있더라도 해당 시트는 적용값과 참조 구조를 파악하는 자료로만 취급한다. 검증 기준으로 자동 승격하거나 외부 기준자료를 대체하지 않는다.

기준값 우선순위는 다음과 같이 표현한다.

1. 사용자가 검증 전에 직접 입력·수정한 기준값
2. 사용자가 별도로 첨부한 외부 제비율표·외부 노임단가표의 파싱 결과
3. 내장 기본값 또는 탐지값. 현재 일반 항목의 기본값은 없으며 부가가치세 10%만 기본값으로 사용할 수 있다.

외부 기준자료가 없거나 파싱에 실패하면 원가계산서 내부 시트 값을 정답으로 사용하지 않고 `기준 없음·검증 불가` 또는 `수기 입력 필요`로 표시한다. 노임단가 비교도 별도 첨부된 외부 노임단가표를 기준으로만 수행한다.

### 원가계산서 파싱과 정규화

파싱 성공 방식을 팀원이 재현 가능한 수준으로 다음 6단계로 설명한다.

1. SheetJS로 첫 시트뿐 아니라 통합문서의 모든 시트와 숨김 여부를 읽는다.
2. 모든 셀을 Workbook IR로 변환하면서 `sheetName`, `address`, `dataType`, `rawValue`, `formula`, `cachedValue`, `displayValue`, `numberFormat`, `mergedRange`, 참조 셀을 보존한다.
3. 시트명과 구조를 이용해 표지·원가계산·공사내역·일위대가·물량·단가·가격조사·노임·제비율·기타 역할로 분류한다.
4. 항목명의 공백과 기호 차이를 정리한 뒤 산재보험료, 고용보험료, 국민건강보험료, 장기요양보험료, 산업안전보건관리비, 간접노무비, 기타경비, 일반관리비, 이윤, 환경보전비, 안전관리비, 부가가치세 등의 표준 항목에 연결한다.
5. 표준 항목이 탐지된 행에서 요율 셀과 금액 셀을 찾는다. 금액은 수식 셀을 우선하고 수식이 없으면 행의 우측 숫자 셀을 사용하며 직접·타 시트 참조를 보존한다.
6. 항목명, 산출기초, 요율, 금액, 수식, 원본 셀, 참조를 정규화된 한 행으로 만들고 검증 결과와 결합한다.

원본 셀 → Workbook IR → 정규화 행 → 검증 기준 연결을 보여주는 설명용 예시 표를 포함한다. 예시 셀 주소와 수치는 실제 검증 파일의 값으로 오해하지 않도록 예시임을 표시한다.

`원가계산서 파싱 성공`은 주요 원가 항목을 표준 행으로 만들고 수식·값·원본 셀·직접 참조를 보존해 검증 화면에 연결했다는 의미다. 직접재료비처럼 여러 시트를 여러 단계로 거치는 전체 계산 계보를 완전히 해석했다는 의미는 아니며, 이 부분은 현재 한계로 함께 적는다.

### 현재 판단

- Excel 참조 오류
- 저장값과 지원 수식 재계산값 비교
- 공급가액 + 부가가치세 = 합계
- 주요 제비율 항목의 요율·금액 비교
- 직종별 노임단가 비교
- 수식 누락과 직접 입력값 경고
- 시트·셀·수식 근거 제시

검증은 수식 연결 여부뿐 아니라 외부 기준표의 비율을 올바르게 적용했는지, 산출기초와 요율을 이용한 기대금액이 실제 금액과 맞는지, 외부 노임단가를 맞게 사용했는지까지 확인한다고 설명한다.

### 현재 출력

- 문서 인식 결과와 Workbook IR JSON
- 오류·확인 필요·주의·정상·검증 불가 상태
- 기대값, 실제값, 차이, 계산과정, 권장 조치
- 원본과 기준자료의 시트·셀 근거
- 정규화 원가계산서 리포트

### 현재 문제

- 제비율표의 병합 다단 헤더와 전체 조건 조합을 정확히 파싱하지 못한다.
- 직접재료비처럼 다른 시트에서 가져오는 원가 항목의 계산 계보를 안정적으로 탐지하지 못한다.
- 원가계산서·제비율표·노임단가표 정규화 검수 화면은 설계와 구현 계획까지만 완료되었다.
- 관련 법령 웹 수집·최신성 확인·해석과 공고문 초안 생성은 아직 구현하지 않았다.

## 5. 시각 디자인

첨부 `6조.html`의 네이비·블루 중심 디자인 언어를 이어간다.

- 네이비 고정 사이드바와 섹션 번호 목차
- 밝은 회색 캔버스와 흰색 문서 카드
- 파랑: 기본 정보와 진행, 초록: 완료, 주황: 진행 중·확인 필요, 빨강: 문제
- 큰 제목과 절제된 카드·표·타임라인 조합
- 외부 폰트·CDN·이미지 서버를 사용하지 않는다.
- 장식적 그래디언트와 반복적인 카드 그리드를 과도하게 사용하지 않는다.

## 6. 화면과 동작

- 데스크톱: 고정 사이드바와 최대 너비 본문
- 모바일: 사이드바를 접고 단일 열 구성, 하단 메뉴 버튼 제공
- 목차 클릭 시 해당 섹션으로 부드럽게 이동
- 스크롤 위치에 따라 현재 목차 항목 강조
- `인쇄/PDF` 버튼으로 브라우저 인쇄 실행
- 인쇄 시 사이드바와 메뉴 버튼을 숨기고 A4 친화적 여백과 줄바꿈 적용
- 실제 앱 화면은 `../qa/upload-desktop.png`, `../qa/recognition-desktop.png`, `../qa/dashboard-desktop.png`, `../qa/detail-desktop.png`를 사용한다.
- 실제 앱 화면은 마우스 클릭 또는 키보드 Enter·Space로 전체 화면에 가까운 모달에서 확대할 수 있어야 하며, Esc·닫기 버튼·배경 클릭으로 닫을 수 있어야 한다.
- 이미지가 없는 환경에서도 대체 텍스트와 캡션으로 내용을 이해할 수 있어야 한다.

## 7. 구현 제약

- 한 개의 독립 HTML 파일 안에 HTML, CSS, JavaScript를 포함한다.
- 외부 패키지, 빌드 과정, 네트워크 요청이 없어야 한다.
- 시맨틱 HTML과 키보드 접근 가능한 링크·버튼을 사용한다.
- `prefers-reduced-motion`을 존중한다.
- 기존 앱 코드와 스타일은 수정하지 않는다.

## 8. 검증 기준

- 1440×900에서 사이드바와 본문이 겹치지 않는다.
- 390×844에서 가로 페이지 넘침 없이 읽을 수 있다.
- 네 개의 실제 화면 이미지가 모두 로드된다.
- 모든 목차 링크가 존재하는 섹션으로 이동한다.
- 인쇄 미리보기에서 사이드바가 사라지고 본문이 잘리지 않는다.
- `6조`, 핵심 소개 문구, 입력·판단·출력, 진행 상황, 두 가지 핵심 미해결 문제와 다음 계획이 빠짐없이 포함된다.
- 원가계산서 6단계 파싱 흐름, Workbook IR 보존 필드, 정규화 예시와 파싱 성공의 한계가 포함된다.
- 외부 제비율표·외부 노임단가표가 판정 기준이며 원가계산서 내부의 동명 시트는 기준이 아니라는 설명이 포함된다.
- 현행 AI 없는 규칙 기반 방식을 유지하고 선택적 AI API만 후속 검토한다는 방침이 포함된다.
- 공고문 생성과 관련 법령 수집·해석이 아직 미구현임을 명확히 표시한다.
- 네 화면 모두 클릭·키보드로 확대되고 모달을 정상적으로 닫을 수 있다.
- 현재 구현과 후속 비전을 혼동시키는 표현이 없다.
